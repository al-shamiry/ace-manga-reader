## Design Context

### Users
A single power-user (the developer) reading manga on their own desktop. Context is leisure reading at a Windows machine, often in a dark room, often for long uninterrupted sessions. The job-to-be-done is "open the app, find where I left off, disappear into the page" — the UI exists only to get out of the way of the artwork. Keyboard and mouse are both first-class; the user knows the shortcuts.

### Brand Personality
**Quiet · focused · immersive.**

The app should feel like a Kindle or Apple Books for manga: calm, confident, and so unobtrusive that after thirty seconds the user forgets the chrome is there. Voice is terse and functional — labels not slogans, status not celebration. No exclamation marks, no marketing copy, no mascots. When the reader is open, the app aspires to be invisible.

Emotional goals: **calm** (never anxious or busy), **trust** (progress is never lost, state is always clear), **focus** (one thing at a time, no nagging).

### Aesthetic Direction
- **Theme**: Dark-first and dark-primary. Two named scales defined CSS-first in `global.css` via `@theme`: `ink-*` (cool 215° tinted neutrals — sumi ink, never pure gray, never `#1a1a1a`) and `jade-*` (175° teal/jade accent). `red-*` is reserved for destructive. All values are OKLCH for perceptual uniformity. Use the named scales directly (`bg-ink-900`, `text-jade-500`) — do not reach for stock `zinc-*` or `indigo-*`. Semantic tokens (`--color-background`, `--color-foreground`, `--color-primary`, etc.) layer on top so a light theme can be added later under `[data-theme="light"]` without compromising dark.
- **Visual tone**: Minimal, native-desktop, content-forward. Generous breathing room around covers and reader pages. Chrome recedes; artwork dominates. **Type pairing**: Newsreader (variable serif, optical-size axis pinned to display) for hero titles via `font-display` / `text-display`; Hanken Grotesk for all UI via `font-sans` (with `ss01`/`cv11` stylistic alts on); `ui-monospace` chain for tabular page counters. Display face is reserved for view-level hero titles — everything else stays in the sans.
- **Iconography**: lucide-solid only, 1.5–2px stroke, consistent sizing per surface (22px in sidenav, 16–18px in toolbars). No emoji, no custom illustrations.
- **Motion**: Purposeful and short (≤200ms). Page flips are direction-aware. Shimmer skeletons for loading. No bouncy springs, no parallax, no decorative animation. Motion communicates state changes — it never performs.
- **Density**: Comfortable but not airy. Manga grids are the densest surface; toolbars and menus are spacious enough to click confidently without feeling like a mobile app.

**References (the right feel)**: Kindle desktop, Apple Books, mpv, native Windows 11 dark surfaces, Plex Desktop's reader-style chrome.

**Anti-references (explicitly avoid)**:
- Generic Material/Bootstrap admin dashboards — no stock card shadows, no default blue, no chip soup.
- Cluttered legacy desktop apps like Calibre — no dense toolbars, no tiny icons, no modal-stacked workflows.
- Childish/kawaii anime aesthetics — no pastel gradients, no mascots, no sparkles, no rounded-everything.
- Web-app feel — no browser-y typography, no link-blue underlines, no "landing page" sections. It must feel like a native app, not a site in a window.

### Design Principles

1. **The page is the product.** Every chrome decision is judged against "does this help the user read manga, or does it get in the way?" When in doubt, hide it, dim it, or move it behind a keystroke. The reader view is sacred — the sidenav already disappears there, and that instinct extends everywhere.

2. **Calm over clever.** Prefer one accent color, one motion duration, one type scale. Restraint is the aesthetic. If a screen feels "designed," it's probably overdesigned. Reach for fewer elements, more whitespace, and quieter contrast before reaching for color or animation.

3. **Keyboard is a first-class input.** The user knows `m`, `f`, arrows, `Esc`. Every new feature should ask "what's the keystroke?" before "what's the button?". Visible affordances exist for discoverability, but the muscle-memory path must always be faster.

4. **Native, not web.** Use semantic tokens, system-feeling spacing, and desktop conventions (right-click menus, hover states, real cursors). Avoid web idioms: hero sections, marketing cards, CTA buttons, link-underlines, breadcrumbs-with-chevrons. This is a Tauri app — lean into it.

5. **Performance is a design value.** Shimmer skeletons appear instantly, scans are cached, animations stay at 60fps, and the reader never blocks on I/O. A slow interaction is a design bug, not just an engineering one. Trade visual flourish for responsiveness every time.
